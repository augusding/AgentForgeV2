🤖 Agent 生成器

为每个岗位生成 V4 四层架构 YAML

输入来源：
IntakeData → roles[].name/responsibilities/experience_tips
IndustryIntel → glossary 注入行业术语
IndustryIntel → methodologies → Agent rules.methodology
IndustryIntel → compliance → Agent soul.boundaries
IndustryIntel → typical_team[].industry_knowledge → Agent soul.beliefs
IndustryIntel → kpis → Agent soul.vibe (关注什么指标)

输出：
agents/*.yaml — 每个 Agent 的完整 V4 配置

👥 Squad 生成器

将 Agent 组织成业务小队

输入来源：
Agent 列表 → 所有已生成的 agent_id
IntakeData → roles[].collaborates_with / reports_to
IndustryIntel → typical_squads 提供行业参考编组
IndustryIntel → kpis → Squad kpi 字段

输出：
squads.yaml — squad_id / name / lead / members / kpi

内部关联：
Squad.lead / members[] 引用 Agent.agent_id
Agent.squad 反向引用 Squad.squad_id

🔄 Workflow 生成器

将业务流程转化为可执行工作流

输入来源：
IntakeData → core_workflows[].name/steps/exceptions
Agent 列表 → 可分配的 agent_id
IndustryIntel → typical_workflows 提供流程参考
IndustryIntel → compliance → 审核步骤 type: "gate"
IndustryIntel → methodologies → step instruction 专业化

输出：
workflows/*.yaml — 每个流程的步骤定义

内部关联：
Step.agent 引用 Agent.agent_id
Profile.sop_triggers 引用 Workflow.name

🧬 Profile 生成器

生成行业 DNA 顶层配置

输入来源：
IntakeData → company_name / communication_style
IndustryIntel → glossary → Profile glossary (合并补全)
IndustryIntel → principles → Profile principles
IndustryIntel → kpis → Profile kpi_definitions
IndustryIntel → compliance → Profile principles 合规条目
Workflow 列表 → Profile sop_triggers

输出：
profile.yaml — 行业标识 / 术语 / 原则 / KPI / 触发器

3. 行业知识包对 Agent V4 四层架构的注入映射 每个字段如何变得"行业化"

Layer 1: Soul（灵魂）— "是谁"

essence (核心定位):
IndustryIntel.typical_team[i].core_duty → 定义角色在该行业的定位

beliefs (核心信条):
IndustryIntel.typical_team[i].industry_knowledge → 注入行业认知
IndustryIntel.methodologies[] → 信条引用行业框架
IntakeData.experience_tips → 客户真实业务经验

vibe (工作风格):
IndustryIntel.kpis[] → 关注什么指标就较真什么

boundaries (边界):
IndustryIntel.compliance[] → 行业合规红线作为"我不做什么"

Layer 2: Rules（规则）— "怎么做"

methodology (工作方法论):
IndustryIntel.methodologies[] → 直接引用行业标准框架
例: 口腔 → "SOAP病历法"，金融 → "KYC/AML框架"
IntakeData.daily_tasks → 结合客户实际工作流程

collaboration (协作接口):
IndustryIntel.typical_team[] → 知道行业中谁和谁协作
IntakeData.collaborates_with → 客户实际组织关系

authority (审批权限):
IndustryIntel.compliance[] → 哪些操作必须审批
IntakeData.decision_authority → 客户定义的权限边界

Layer 3: Identity（身份）— 展示层

display_name / role_tag:
IndustryIntel.typical_team[i].role → 使用行业通用称呼
例: 口腔用"种植医生"而非"操作员"

avatar_emoji / color:
由 LLM 根据角色特征自动选择，与行业知识关联不大

Layer 4: Config（配置）— 技术层

skills[]:
IndustryIntel.methodologies[] → 技能对应行业方法论
IntakeData.skills_required → 客户要求的技能

delegation.can_access[]:
IndustryIntel.typical_squads[] → 同组 Agent 可互访

capabilities.tools[]:
根据角色职责从已注册工具中匹配

llm.temperature:
创意岗 (内容/设计) → 0.7，分析岗 (数据/风控) → 0.3
4. 新旧方案生成效果对比 以"口腔诊所"行业的种植医生为例

❌ 旧方案（无行业情报）

agent_id: "implant-doctor"
soul:
  essence: |
    我是一名专业的医生，负责种植牙的方案设计
    和手术执行。对患者负责，追求高质量的治疗效果。
  beliefs:
    - "患者安全是第一位的"
    - "持续学习，跟进最新技术"
    - "团队协作很重要"
  boundaries:
    - "不做超出能力范围的手术"
rules:
  methodology: |
    1. 患者检查
    2. 制定方案
    3. 执行手术
    4. 术后跟踪

▲ 通用、空洞、缺乏行业深度
  beliefs 像鸡汤，methodology 像流水账
  任何行业的任何医生都能套用

✅ 新方案（行业情报驱动）

agent_id: "implant-doctor"
soul:
  essence: |
    种植不是拧螺丝，是在活体骨组织里做精密工程。
    每一颗种植体的角度、深度、扭矩都影响十年后的效果。
  beliefs:
    - "CBCT是种植的眼睛，没有三维影像的种植方案是盲人手术"
    - "骨量不足时宁可先做GBR再延期种植，不冒即刻种植的风险"
    - "椅旁时间不是越短越好，该慢的步骤一步都不能省"
    - "数字化导板能把种植精度从毫米级提升到亚毫米级"
  boundaries:
    - "全身系统性疾病未控制的患者，必须转诊不得手术"
    - "超出执业许可范围的正颌手术，转三甲口腔医院"
rules:
  methodology: |
    种植全流程：
    1. 初诊评估：全口CBCT + 口内扫描 + 全身健康筛查
    2. 方案设计：3D种植规划软件，确定植体型号/角度/深度
    3. 导板制作：数字化种植导板，术前验证就位精度
    4. 一期手术：翻瓣/不翻瓣 + 植体植入 + 扭矩值记录
    5. 愈合期管理：3-6个月随访，骨结合评估
    6. 二期修复：取模/口扫 + 基台选择 + 冠修复
    7. 长期维护：每年复查CBCT，定期洁治

▲ 专业、有深度、有态度
  beliefs 有行业锋芒，methodology 是真实临床流程
  只有口腔种植医生才会这样说话
